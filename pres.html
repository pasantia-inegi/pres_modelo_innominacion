Problemática (1/)

  • Por ley de transparencia activa, los órganos y servicios gubernamentales deben publicar las funciones administrativas del personal de planta, a contrata y el que se desempeñe en virtud de un contrato de trabajo, y contratadas a honorarios.

  • En el caso del personal con contrato a honorarios, estos deben realizar mensualmente un informe de cumplimiento, en donde se describen las actividades realizas durante el mes.

  • En estos documentos, existe información confidencial de las personas: el RUN (Rol Único Nacional) y firma, los cuales deben ser censurados en la publicación.

Modelamiento | Innominación de informes (1/)

Una vez entrenado el modelo, la metodología para realizar la innominación sigue el siguiente flujo:

  • Paso 1: Procesar informe

    • Transformar el formato de pdf a imagen.

  • Paso 2: Identificar RUN y firmas

    • Para cada conjunto de imágenes de un informe, se identificaran las firmas y el run.

    • Una vez identificada la sección donde están los objetos, se ocultarán las secciones con un rectángulo negro.

    • En cada uno de los informes, en total debe existir un run y 2 firmas. En el caso de no cumplir esta condición, se etiquetará el documento como informe dudoso y se registrará para que el equipo encargado realice la revisión manual.

  • Paso 3: Transformar el informe a su formato original

    • Realizar unión de imágenes y generar pdf.

Empaquetado | Estructura

Empaquetado | Estructura: Flask

  • Flask es un microframework para desarrollo web de backend, famoso por ser sencillo, liviano y flexible. Es la base de otros frameworks más complejos como FastAPI.

  • Flask funciona como backend de la aplicación, comunicándose con el frontend y manejando las lógicas de la aplicación: subida y extracción de informes, manejo de errores y éxitos, gestión de requests del cliente.

  • Una vez extraidos los informes, ocurren dos cosas: se da inicio a la task que se encarga de anonimizar los informes y se le da fin al request, idealmente, con un mensaje de éxito.

  • La task es manejada por Celery y Redis

Empaquetado | Estructura: Celery + Redis

  • Celery es una cola de tasks asíncrona y distribuida.

  • Redis funciona como broker y backend de Celery, funcionando como data store in-memory.

  • Una vez comienza la task, la conexión con el cliente se corta y Celery se encarga de ejecutarla:

    • Obtiene las predicciones para cada informe.
    • Guarda los resultados en MinIO.
    • Envía un mail de notificación con link de descarga.

  • En este contexto, Redis como broker se encarga de la comunicación entre Flask y Celery y como backend se encarga de guardar los resultados y el estado actual de la tarea.

En este caso utilizamos un solo trabajador a la vez, por lo que no estamos utilizando todas las capacidades de Celery.

Empaquetado | Estructura: MinIO

  • MinIO actúa como capa de almacenamiento, por lo que aloja los archivos .zip que contienen los informes anonimizados.
  • Además, nos permite compartir un link de descarga que expira luego de un tiempo fijado por nosotros.
